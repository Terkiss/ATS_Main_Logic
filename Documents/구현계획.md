# 구현 계획 (Implementation Plan) - Milestone 7: Real-time Tick & State Sync

## Phase
Phase 2 — Game Server Edition (첫 번째 마일스톤)

## 개요
실시간 게임 서버의 핵심인 고정 주기 Tick Loop를 도입하고, 게임 상태 스냅샷 관리 및 Delta Broadcast를 구현합니다.
현재 서버는 수신 이벤트 기반(Reactive)으로만 동작하므로, 능동적으로 게임 상태를 갱신하고 전송하는 구조를 추가합니다.

## 세부 작업 항목

### 1. 서버 Tick Loop 구현
- **목표**: 고정 주기(20Hz/50ms 기본, 설정 가능) 기반의 `GameLoop` 백그라운드 스레드 도입
- **구현 내용**:
  - `IGameLoop` 인터페이스 (Start, Stop, RegisterTickHandler, TickRate 프로퍼티)
  - `GameLoop` 구현체: 고정 간격 루프 + Stopwatch 기반 정밀 타이밍
  - DI를 통한 주입 및 Program.cs 등록
  - Tick 콜백 시스템: Logic Plugin이 Tick 핸들러를 등록 가능

### 2. 게임 상태 스냅샷 구조 설계
- **목표**: Tick 단위로 관리되는 게임 상태 모델 정의
- **구현 내용**:
  - `GameEntity` 모델 (EntityId, Position, Rotation, Velocity, State 등)
  - `WorldState` 모델 (TickNumber, Timestamp, Entities 딕셔너리)
  - `RoomState` 모델 (RoomId, 참여자 목록, WorldState 참조)
  - 스냅샷 링 버퍼: 과거 N 프레임 스냅샷 보관 (M8 Lag Compensation 대비)

### 3. Delta Broadcast 구현
- **목표**: 매 Tick마다 변경된 엔티티만 클라이언트에 전송
- **구현 내용**:
  - `DeltaCalculator`: 이전 스냅샷과 현재 스냅샷을 비교하여 변경 목록 생성
  - `StateSyncProtocol` 프로토콜 추가 (ProtocolSelect enum)
  - 직렬화 최적화: 변경된 필드만 포함하는 Delta 패킷 구성

### 4. 브로드캐스트 최적화
- **목표**: P2PGroup 기반 룸 단위 멀티캐스트
- **구현 내용**:
  - `IRoomBroadcaster` 인터페이스: 룸 내 클라이언트에 패킷 전송
  - P2PGroup.Members를 활용한 타겟 전송
  - IMessageSender 확장: 멀티캐스트 메서드 추가

### 5. 입력 큐 구조
- **목표**: 클라이언트 입력을 Tick 단위로 묶어 처리
- **구현 내용**:
  - `InputQueue<T>` 클래스: ConcurrentQueue 기반 입력 버퍼
  - `GameInputProtocol` 프로토콜 추가
  - Tick 처리 시 큐에서 입력을 꺼내 게임 로직에 적용

## 완료 기준
- 서버 Tick이 목표 Hz에서 ±2ms 오차 이내로 안정적으로 동작
- 10개 이상의 클라이언트가 동일 룸에서 Delta Sync 수신 성공
