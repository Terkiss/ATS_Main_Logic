# 구현 계획 (Implementation Plan) - Milestone 4: Plugin Ecosystem & Hot-reload

## 개요
TeruTeruServer의 로직 플러그인 생태계를 완성하는 단계입니다.
현재 `PluginManager`가 DLL 동적 로드를 지원하지만 Hot-reload는 부분적이며(이전 AssemblyLoadContext 언로드 없음), 플러그인 격리·의존성 관리·문서화가 부재합니다.
이 마일스톤을 완성하면 Logic 개발과 엔진 개발을 완전히 독립적으로 진행할 수 있습니다.

## 세부 작업 항목

### 1. 무중단 플러그인 교체 (Hot-reload 완성)
- **목표**: 서버 재시작 없이 DLL 교체 시 이전 로직 언로드 → 신규 로직 로드 사이클 완성
- **구현 내용**:
  - `PluginManager.ReloadPlugins()`에서 이전 `AssemblyLoadContext`를 명시적으로 `Unload()` 처리
  - GC 압박 없이 안전한 컨텍스트 전환을 위한 `WeakReference` 기반 언로드 확인
  - `FileSystemWatcher` 이벤트 debounce 처리 (중복 트리거 방지)

### 2. 플러그인 간 의존성 관리
- **목표**: 플러그인 A가 플러그인 B의 서비스를 참조할 수 있는 구조 허용
- **구현 내용**:
  - `IPluginDependency` 인터페이스 도입 (SDK에 정의)
  - `PluginManager`가 여러 DLL을 로드하고 의존성 순서를 자동 해결하는 로직 추가
  - 순환 참조 감지 및 오류 로깅

### 3. 어트리뷰트 자동 문서화
- **목표**: `[Protocol]` / `[Rpc]` 어트리뷰트 정보를 런타임 메타로 추출하여 API 목록 자동 생성
- **구현 내용**:
  - 플러그인 로드 시 리플렉션으로 `[Protocol]`, `[Rpc]`, `[RequiresAuth]` 어트리뷰트를 스캔
  - `PluginMetadata` 모델에 수집하여 콘솔 명령 또는 API로 조회 가능하게 노출

### 4. 플러그인 샌드박스 (오류 격리)
- **목표**: 비정상 플러그인이 엔진 크래시를 유발하지 않도록 격리
- **구현 내용**:
  - `LogicProxy`의 각 위임 호출을 `try-catch`로 감싸서 플러그인 예외가 서버 루프를 죽이지 않도록 방어
  - 연속 N회 이상 예외 발생 시 해당 플러그인을 자동 비활성화하고 경고 로그 출력
  - 비활성화된 플러그인 상태를 조회할 수 있는 관리 인터페이스 추가

### 5. SDK NuGet 패키징
- **목표**: `TeruTeruServer.SDK`를 외부 개발자가 참조 가능한 NuGet 패키지로 배포 준비
- **구현 내용**:
  - `TeruTeruServer.SDK.csproj`에 NuGet 메타데이터(PackageId, Version, Description, Authors) 추가
  - `dotnet pack` 명령으로 `.nupkg` 생성 검증
  - XML 문서 주석 빌드 시 자동 포함 설정

## 검증 계획
1. **Hot-reload 검증**: 서버 운영 중 DLL 교체 후 신규 `[Protocol]` 핸들러가 즉시 동작하는지 확인.
2. **오류 격리 검증**: 의도적으로 예외를 발생시키는 플러그인 로드 시 엔진이 종료되지 않고 오류 로그만 기록되는지 확인.
3. **문서화 검증**: 플러그인 로드 후 등록된 프로토콜/RPC 목록이 정확히 출력되는지 확인.
4. **NuGet 패키징**: `dotnet pack`으로 `.nupkg` 파일이 정상 생성되는지 확인.
5. **릴리스 파이프라인**: `verify-release.sh` 100% 통과.

## 다음 작업 안내
이후 Milestone 4가 성공적으로 구현되면, 작업자는 **Milestone 5: Scalability & Clustering** 로 진입하여 수평 확장 구조를 구축하게 됩니다.
