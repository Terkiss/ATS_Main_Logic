# 구현 계획 (Implementation Plan) - Milestone 2: Security Hardening

## 개요
TeruTeruServer의 두 번째 마일스톤인 **보안 레이어 완성**을 목표로 합니다. 현재 설계 수준인 AES/Seed 암호화와 JWT 인증 구조를 실제 공격 벡터(Replay Attack, DoS)에 대응할 수 있도록 강화하고, 플러그인 생태계를 위한 암호화 추상화를 진행합니다.

## 세부 작업 항목

### 1. JWT Refresh Token 흐름 구현
- **목표**: Access Token 만료 시 안전한 재발급 사이클 설계
- **구현 내용**:
  - `LoginProtocol` 응답에 `RefreshToken` 필드 추가
  - `TokenRefreshProtocol` 신규 메시지 추가 및 `LogicPlugin` 연동
  - Access Token과 Refresh Token의 분리된 만료 시간(예: Access 2시간, Refresh 7일) 적용

### 2. Rate Limiting 미들웨어 추가
- **목표**: 악의적인 패킷 플러딩(DoS) 방어를 위한 연결별 요청 속도 제한
- **구현 내용**:
  - `RateLimitMiddleware` 신규 구현
  - `ClientSession`에 최근 처리 패킷 수 추적 및 1초당 허용 한도(예: 50개) 초과 시 해당 패킷을 무시(Drop)하여 처리 부하 방지

### 3. Replay Attack 방어
- **목표**: 이미 처리된 패킷을 캡처하여 재전송하는 공격 차단
- **구현 내용**:
  - 패킷 헤더에 4바이트 `SequenceNumber` 필드를 추가하여(브레이킹 체인지 감수) JSON 파싱 이전에 가장 빠른 단계에서 재전송 공격 차단
  - 기존 클라이언트 SDK(`TeruClient`) 및 `DummyClient`의 패킷 빌드/파싱 로직 동기화 업데이트

### 4. AuthMiddleware 강제화 (`[RequiresAuth]`)
- **목표**: 하드코딩된 예외 처리를 제거하고 태그 기반 보안 구조 확립
- **구현 내용**:
  - `[RequiresAuth]` 커스텀 어트리뷰트 도입
  - `RoutingMiddleware` 등 라우팅 전 단계에서 타겟 프로토콜/RPC의 태그를 검사하여 토큰이 없으면 패킷 드롭
  - 기존 `AuthMiddleware`는 토큰 파싱 후 `Session` 상태만 마킹하도록 역할 축소

### 5. Pluggable Crypto Interface
- **목표**: 암호화 알고리즘 교체의 용이성 확보 (Seed, AES 등)
- **구현 내용**:
  - `ICryptoService` 인터페이스 정의 (`Encrypt`, `Decrypt` 메서드)
  - 기존 `SeedEncrypt`를 `SeedCryptoService`로 래핑하여 DI(의존성 주입) 가능하게 구성
  - `DecryptionMiddleware` 등 파이프라인에서 인터페이스 기반으로 동작하도록 변경

## 검증 계획
1. **토큰 검증**: `LoginProtocol`을 통해 Access/Refresh 토큰이 발급되고, 만료 후 갱신이 정상적으로 이루어지는지 테스트.
2. **Rate Limit 차단**: 의도적으로 초당 한계치 이상의 패킷을 서버로 전송 시 서버가 연결을 끊고 방어하는지 테스트.
3. **권한 제어 차단**: `[RequiresAuth]`가 달린 프로토콜을 토큰 없이 전송 시 처리가 거부되는지 테스트.
4. **암호화 인터페이스**: 플러그인 구조로 암호화/복호화가 기존과 동일하게 동작하는지 검증.
5. **릴리스 파이프라인**: 모든 구현 후 `verify-release.sh` 100% 통과 여부 재확인.
