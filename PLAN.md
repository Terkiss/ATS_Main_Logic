# Mobile Network Handover & P2P Resilience Plan

## 1) 목표와 범위
- Wi-Fi ↔ 셀룰러 전환, 일시적 끊김, NAT 변경 상황에서 "세션 유지 + 빠른 복구"를 달성한다.
- P2P 연결은 "가능하면 직접, 실패하면 서버 릴레이" 전략을 기본으로 한다.
- 기존 프로토콜 정의 자체는 크게 바꾸지 않고, 재접속·상태관리·관측성을 우선 강화한다.

## 2) 현재 구조 요약 (점검 결과 반영)
- 서버는 단일 TCP 리스너와 소켓별 `ReceiveAsync` 루프 기반이다.
- `HostID`, `GameID`, `ClientSession` 매핑은 이미 존재해 재접속 설계에 유리하다.
- 반면, "재접속 시 기존 세션에 소켓을 재바인딩"하는 경로와 "유예시간(grace period)" 개념이 없다.
- 연결 끊김 시점에 곧바로 세션이 정리되기 때문에 모바일 전환에 취약하다.

## 3) 핵심 설계 원칙
1. 세션과 연결을 분리한다.
   - 세션(Session)은 유지되고, 연결(Socket)은 교체 가능해야 한다.
2. 끊김 즉시 제거 대신 유예시간을 둔다.
   - 일시 단절은 복구 기회를 주고, 장기 단절만 정리한다.
3. 재접속은 "동일 세션 복원"이 기본이다.
   - 새 접속이 오더라도 세션 토큰이 유효하면 소켓만 교체한다.
4. P2P는 상태 기반으로 관리한다.
   - Signaling, Direct(P2P), Relay(Server) 상태를 명시적으로 둔다.

## 4) 단계별 실행 계획

### Phase A. 세션 상태 모델 보강
- `ClientSession`에 다음 상태를 추가한다.
  - `LastSeenUtc` (마지막 정상 통신 시각)
  - `ConnectionState` (Connected / Grace / Disconnected)
  - `ReconnectToken` (재접속 검증용)
  - (선택) `P2PState` (Signaling / Direct / Relay)
- 세션 생성 시 `ReconnectToken`을 발급하고 서버 메모리에 보관한다.

완료 기준:
- 세션이 "소켓 유무"와 무관하게 독립적으로 상태를 가질 수 있다.

### Phase B. 끊김 처리: Grace Period 도입
- 끊김 감지 시 즉시 삭제하지 않고 다음처럼 처리한다.
  1) 세션 상태를 `Grace`로 전이
  2) `LastSeenUtc` 갱신
  3) 소켓은 종료하되, 세션은 유지
- 별도 정리 루프에서 `GraceTimeout`을 초과한 세션만 최종 제거한다.

권장 기본값:
- `GraceTimeout`: 30~90초 (모바일 전환 고려)

완료 기준:
- 짧은 단절은 세션이 유지되고, 일정 시간 이후에만 제거된다.

### Phase C. 재접속 경로: "소켓 교체" 구현
- 재접속 요청 시 다음을 만족하면 동일 세션 복원:
  - `HostID` 또는 `GameID` + `ReconnectToken`이 유효
- 복원 절차:
  1) 기존 소켓이 있으면 종료
  2) `players[HostID]`를 새 소켓으로 교체
  3) 세션 상태를 `Connected`로 전이
  4) 필요한 초기 동기화 응답 전송

완료 기준:
- 네트워크 전환 후 "같은 세션으로 복귀"가 가능하다.

### Phase D. P2P 전략 정리 (서버 관점)
- 서버는 최소한 아래 정보를 세션에 저장/관리한다.
  - 최근 P2P 성공 여부
  - 실패 횟수와 마지막 실패 시각
  - 현재 전송 모드(Direct/Relay)
- 정책 예시:
  - P2P 실패가 연속 N회이면 일정 시간 릴레이 고정
  - 재접속 직후에는 일단 Relay로 복구 후 백그라운드에서 P2P 재시도

완료 기준:
- P2P 성공/실패와 무관하게 서비스 연속성이 유지된다.

### Phase E. 관측성/운영 편의 강화
- 다음 이벤트에 구조화된 로그를 추가한다.
  - 끊김 감지, Grace 진입, Grace 만료, 재접속 성공/실패, 소켓 교체
- 운영 명령(콘솔 커맨드) 추가 아이디어:
  - `session_list`: 세션 상태/마지막 통신 시각 출력
  - `session_kick <hostId>`: 강제 종료
  - `grace_stats`: grace 상태 통계 출력

완료 기준:
- "왜 끊겼고 왜 복구가 안 됐는지"를 로그로 추적 가능하다.

## 5) 테스트 시나리오

### 필수 시나리오
1) 정상 접속 → 데이터 송수신 → 유지
2) Wi-Fi 차단 5~15초 → 복구 → 동일 세션 유지
3) Wi-Fi → 셀룰러 전환 → 재접속 → 동일 HostID 유지
4) 장시간 단절(GraceTimeout 초과) → 세션 정리
5) 재접속 토큰 불일치 → 복구 거부

### 권장 계측
- 재접속 성공률
- Grace 상태 진입 횟수 대비 복구율
- 평균 복구 시간 (MTTR)

## 6) 구현 순서 제안 (리스크 최소화)
1) 세션 상태 필드 추가 (Phase A)
2) Grace 도입 (Phase B)
3) 소켓 교체형 재접속 (Phase C)
4) 관측성 강화 (Phase E)
5) P2P 정책 정교화 (Phase D)

## 7) 기대 효과
- 모바일 네트워크 전환에서 "즉시 탈락" 대신 "복구 우선" 동작
- 세션 연속성 확보로 사용자 경험 개선
- 운영 중 장애 원인 파악이 쉬워짐
